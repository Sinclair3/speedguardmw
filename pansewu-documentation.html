Section 1

System Overview

Pansewu (Chichewa: "on the road") is a free, browser-based road safety application purpose-built for the M1 highway in Malawi — the 306 km national trunk road connecting Karonga in the north to Nsanje in the south, passing through Lilongwe and Blantyre.

1.1 Purpose & Problem Statement

The M1 corridor has one of the highest road traffic accident rates in sub-Saharan Africa. Drivers face:

  • Unannounced mobile speed traps and police checkpoints
  • No real-time visibility of road hazards ahead
  • Limited access to emergency services or nearest hospital locations
  • No coordinated communication between drivers on the route

Pansewu addresses these gaps through crowd-sourced hazard reporting, live GPS speed monitoring, and a regional driver communication channel — all delivered as a zero-install PWA that works on any smartphone.

3.2 Real-Time Data Flow

All live features — driver positions, chat, trap votes — use a single Supabase Realtime channel (m1_drivers) via WebSocket broadcast. There is no polling loop for live data.

EventDirectionChannelPayload
Driver position updateDevice → Allm1_drivers broadcastid, lat, lng, speed, zone, nickname
Chat messageDevice → Allm1_drivers broadcasttype:chat, text, zone, nickname, replyText
Trap voteDevice → Allm1_drivers broadcasttype:vote, trapId, isYes
Trap verified pushSupabase DB → Edge Fn → Push Service → DeviceDatabase webhookyes_votes ≥ 3 triggers notify-trap

3.3 Offline Strategy

The service worker (sw.js) provides a cache-first strategy for static assets. Map tiles are cached on first load. The following features work without internet:

  • GPS speed display (device GPS only)
  • Hardcoded trap markers and road service locations
  • SOS call (direct tel: link — no data needed)
  • Map base tiles (served from cache)
  • All Settings, Score, and Trip History (localStorage)

Community traps, live chat, driver positions, and push notifications require an active data connection.

4.2 Row Level Security (RLS)

All tables have RLS enabled. The anon key used in the frontend can:

  • sg_traps: SELECT (all), INSERT (own device_id), UPDATE yes_votes/no_votes only, DELETE own rows only
  • sg_chat: SELECT (last 2h, zone-filtered), INSERT (authenticated by anon key), no UPDATE or DELETE by others
  • sg_push_subs: INSERT and UPDATE own device_id rows only, no SELECT of other devices
  • sg_notifications: SELECT only (service worker reads, Edge Function writes with service role key)

4.3 Edge Function — notify-trap

A Deno-based serverless function triggered by a Supabase Database Webhook when sg_traps.yes_votes reaches 3. It:

  1. Checks a 24-hour cooldown (last_notified) to prevent spam
  2. Queries sg_push_subs for subscribers in the matching zone
  3. Updates last_notified on the trap record
  4. Inserts the notification text into sg_notifications
  5. Sends an empty Web Push to each subscriber endpoint using pure SubtleCrypto (no npm dependencies)
  6. Removes stale subscriptions that return HTTP 410 Gone
// notify-trap invocation flow
DB Webhook (yes_votes = 3)
  → Edge Function (Deno)
    → Check cooldown
    → Query sg_push_subs WHERE zones @> [zone]
    → VAPID JWT (ES256 via SubtleCrypto)
    → POST empty push to each endpoint
    → INSERT into sg_notifications
    → Device Service Worker wakes up
    → SW fetches sg_notifications → shows notification

Section 6

Core Features — Technical Detail

6.1 GPS & Speed Tracking

GPS is acquired via the browser's navigator.geolocation.watchPosition() API with enableHighAccuracy: true. Position updates call updateUI(lat, lng, speed) which:

  • Updates the speed display with colour coding (green/yellow/red)
  • Moves the map marker and re-centres if auto-follow is on
  • Calculates trip progress using latProg(lat) — a linear interpolation of the M1's latitude range
  • Checks proximity to all known traps via Haversine distance (hav())
  • Updates the safe driver score (1 point per 10s within limit, -2 per 10s over)
  • Broadcasts position to Supabase Realtime channel (throttled to once per 5s)

Zone Detection

The M1 is divided into three zones based on kilometre position (derived from latitude):

ZoneKm RangeCoverage
North0 – 399 kmKasumulu Border → Nkhotakota
Central400 – 699 kmNkhotakota → Lilongwe → Dedza
South700 – 1140 kmDedza → Blantyre → Nsanje

6.2 Speed Trap System

Two types of traps exist in the system:

Hardcoded Traps

Defined in the var traps = [...] array in index.html. These are permanent, verified locations (toll gates, known checkpoints). They are never deleted by voting and are always shown on the map. Each entry has: id, name, lat, lng, limit, type, subtype, km, verified.

Community Traps

Submitted by drivers via the + Trap button, stored in sg_traps on Supabase. Lifecycle:

  1. Driver reports trap → POST to sg_traps with yes_votes=0
  2. Other drivers vote 👍 → yes_votes increments via PATCH
  3. At yes_votes = 3 → trap rendered as "verified", push notification triggered
  4. At no_votes = 3 → trap auto-deleted from Supabase and removed from map
  5. All community traps expire after 2 hours regardless of vote count

6.3 Regional Chat

Chat operates over a Supabase Realtime WebSocket channel (m1_drivers) using the broadcast mode. Key design decisions:

  • No server-side zone filtering — all messages go to all connected clients; zone filtering happens client-side in receiveChatMsg()
  • Outgoing zone stamping — the sender's active region tab (chatRegionTab) is stamped on outgoing messages. If on "All M1" with no GPS, sending is blocked
  • History — last 50 messages from the past 2 hours fetched from sg_chat on open, filtered by zone
  • Self-exclusion — device ID checked on receive to prevent echo (if(pl.id === myId) return)
  • Mentions — @nickname parsed client-side; mentioned driver gets stronger visual + audio notification
  • Deletion — broadcast delete event marks message as deleted on all devices simultaneously

6.4 SOS Emergency System

The SOS button is a direct tel: link requiring no internet. Flow:

  1. User taps 🆘 SOS → confirmation modal shown (prevents accidental trigger)
  2. On confirm → window.location.href = 'tel:' + sosContact.phone
  3. Device dials the saved emergency contact
  4. App displays current GPS coordinates for verbal reporting

Contact is stored in sg_sos localStorage. No server is involved — works completely offline.

6.5 Push Notifications

Uses the standard Web Push API with VAPID (Voluntary Application Server Identification) authentication. No Firebase or third-party push service.

  1. User grants notification permission in browser
  2. App calls registration.pushManager.subscribe({userVisibleOnly:true, applicationServerKey:VAPID_PUBLIC_KEY})
  3. Subscription (endpoint + keys) saved to sg_push_subs with selected zones
  4. When a trap reaches 3 yes_votes, a Supabase Database Webhook fires notify-trap
  5. Edge Function sends an empty push (no encrypted payload) to save bandwidth
  6. Service worker wakes up, fetches latest entry from sg_notifications, shows the notification

6.6 Safe Driver Score

A gamified scoring system tracked entirely in localStorage — no server-side score storage.

ActionPoints
10 seconds driving within speed limit+1
10 seconds driving over speed limit-2
Submitting a trap report+5
Your trap gets verified (3 votes)+10
Voting on a trap (yes or no)+2
Completing a trip with destination+15

Score tier badges: 🥇 Gold (≥500), 🥈 Silver (≥200), 🥉 Bronze (<200).

Section 7

Admin Panel

The admin panel (admin.html) is a separate page accessible only by the developer. It is not linked from the main app and has no authentication — security relies on obscurity of the URL. A proper authentication layer should be added before any public disclosure of the admin URL.

7.2 Route Simulator — SIM_ROUTE

The simulator uses a SIM_ROUTE array of waypoints covering the full M1. Each waypoint is [lat, lng, name, km]. There are 56 waypoints from Kasumulu Border (0 km) to Nsanje (1140 km). The simulator interpolates between waypoints at a configurable speed (20–120 km/h).

Key simulator variables

Current waypoint index — default 34 (Lilongwe)
simFrac
0–1 fraction of progress between simIdx and next waypoint
simDir
1 = northbound→south, -1 = south→north
SIM_TICK_MS

56 waypoints

7.3 Admin Access

⚠️
The admin panel has no login screen. Access is controlled only by knowing the URL. For production hardening, add HTTP Basic Auth at the GitHub Pages level or restrict via a Cloudflare Access policy in front of the domain.

Security Considerations

9.1 Credential Exposure

CredentialIn Frontend?Risk LevelMitigation
SUPA_KEY (anon)YesLowAnon key is publishable; RLS policies restrict all access
VAPID_PUBLIC_KEYYesNonePublic by design — required for push subscription
VAPID_PRIVATE_KEYNo (Edge Fn only)High if leakedNever committed to repo; stored in Supabase secrets
Service Role KeyNo (Edge Fn only)Critical if leakedNever committed to repo; stored in Supabase secrets

9.2 Privacy

  • No user accounts — identity is a random UUID in localStorage; nothing is linked to a real person
  • Location rounding — chat messages only store lat/lng rounded to 2 decimal places (~1 km precision)
  • Driver positions — sent to Realtime channel but not persisted to database
  • Chat history — deleted after 2 hours; no long-term message storage
  • No cookies — all state in localStorage; no tracking

9.3 Content Security

  • Chat input is escaped via escChat() before DOM insertion to prevent XSS
  • All Supabase API calls use parameterised query strings — no raw SQL from client
  • Trap submission rate is limited implicitly by the vote system — a single device cannot verify its own report

9.4 Known Security Gaps

🔒
Admin panel is unauthenticated. Anyone with the URL can access admin.html. This should be addressed before the URL becomes publicly known.

No rate limiting on trap reports. A malicious device could flood sg_traps with fake reports. Consider adding a Supabase RLS policy limiting inserts per device_id per hour.
